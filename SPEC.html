1

Overview & Problem Statement

1.1 The Problem

The user applies to jobs on LinkedIn and Bayt. Saved jobs are applied to one by one. Two application paths exist:

✓ Easy Apply (low pain)

In-platform apply with a few quick questions. Fast, low friction — not the target problem.

✗ External Apply (high pain)

Redirect to employer's own ATS. Long repetitive forms: experience dates, education dates/scores, certs, skills, custom questions. Auto-fill is incomplete. Slow and tedious for every application.

1.2 The Solution

A Chrome browser extension that, on demand, reads the application form on the currently open page and fills it using a locally stored markdown knowledge base (the "profile"). Filling is hybrid: deterministic rules handle obvious fields; a configurable LLM handles ambiguous fields and drafts custom answers. The user always reviews before submitting — the extension never submits on the user's behalf.

1.3 Value Delivered

  • Eliminate repetitive manual entry of experience / education / cert data.
  • One maintained profile, reused across unlimited forms.
  • Multiple tailored profiles for different target roles.
  • Local application history to avoid duplicate applications.

  • N9Cloud sync / multi-device. All data is local.
  • N10Cover letters. No generation or storage; handled manually.

  • Technically comfortable: can install a dev extension, obtain LLM API keys, edit markdown.

5.1 Component Responsibilities

Side Panel (React): all user-facing UI — profile management, fill/review table, Q&A review, settings, history. Opened by clicking the extension icon. Also performs folder picks and permission re-grants (requires user activation — see §5.2).

Background Service Worker: the orchestrator. Owns the fill state machine, LLM provider layer + fallback, history/dedupe, and file I/O on already-permitted handles. The only context that holds API keys at runtime. MV3 service workers are ephemeral — see EC-22.

Content Script: injected into the active tab on user action. Detects fields, writes values with correct DOM events, flags upload fields, hosts ATS adapters. Holds no secrets and makes no network calls.

Options Page: advanced configuration (provider keys, fallback order, data-folder selection, defaults).

5.2 Why This Shape

File System Access permission must originate in the side panel. showDirectoryPicker and requestPermission require user activation, which a service worker does not have. The side panel owns folder picks and re-grants. An MV3 service worker can read/write an already-permitted handle from IndexedDB, but cannot raise a permission prompt. The FileStore abstraction (§7.5) hides which context performs I/O.
Keeping the content script free of secrets and network access limits the blast radius if a job page is hostile.

Data Model & Knowledge Base

7.1 Folder Layout

<data-folder>/
├── profiles/
│   ├── ml-engineer.md
│   ├── data-analyst.md
│   └── ...              # one .md file per named profile
├── qa-bank.md            # shared Q&A bank (D20)
├── history.json          # application history
└── settings.json         # non-secret settings
Secrets are NOT stored here. LLM API keys live in chrome.storage.local (§15.3), never in the plaintext folder the user might sync or share.

7.2 Profile Markdown Schema

Human-readable markdown with YAML frontmatter and well-known section headings. Parser is tolerant: unknown sections are preserved; missing sections are treated as empty.

---
profile_name: ML Engineer
target_role: Machine Learning Engineer
updated: 2026-06-03
---

# Personal Information
- Full name: <your-full-name>
- Email: <your-email>
- Phone: +92-3xx-xxxxxxx
- Location: City, Country
- LinkedIn: https://linkedin.com/in/...
- GitHub: https://github.com/...

# Work Experience
## Senior ML Engineer — Acme Corp
- Start: 2022-03
- End: present          # "present"/"current" => currently employed
- Employment type: Full-time
- Highlights:
  - Did X, improving Y by Z%.

# Education
## BSc Computer Science — Example University
- Start: 2015-09
- End: 2019-06
- Grade: 3.7/4.0        # stored as written; mapper reformats per field

# Certifications
## AWS Certified Machine Learning – Specialty
- Issuer: Amazon Web Services
- Issued: 2023-05
- Expires: 2026-05

# Skills
- Languages: Python, SQL, TypeScript
- ML: PyTorch, scikit-learn
- Cloud: AWS, GCP

# Preferences
- Work authorization: ...
- Notice period: 1 month
- Willing to relocate: Yes
- Salary expectation: ...

Date convention: store as YYYY-MM or YYYY-MM-DD; use present/current for ongoing roles. Mapper reformats to whatever the target field expects (§11.4).

7.5 Persisting File System Access

  • On first run the user picks the data folder (user gesture in the side panel).
  • The FileSystemDirectoryHandle is stored in IndexedDB (handles are structured-cloneable; chrome.storage cannot store them).
  • On later sessions: call queryPermission. If not granted, call requestPermission from the side panel (needs user activation); show "Reconnect data folder" prompt.
  • All reads/writes go through a FileStore abstraction — caller doesn't know which context does I/O.

⚠ Write Safety (§7.5.1)

  • Re-read the target file immediately before writing.
  • For append-style changes (Q&A save-back): append to freshly read content.
  • For full-rewrite (profile save from F1.3): show diff/preview against on-disk content; if file changed since load, surface conflict instead of clobbering.
  • Writes SHOULD be atomic (temp file + rename) to avoid half-written state on failure.

Data File Versioning (§7.5.2)

history.json and settings.json carry a version field. On load, if the on-disk version is older, run a forward migration (back up as *.bak). An unknown/newer version is treated read-only with a warning.

Functional Requirements

MUST/SHOULD/MAY per RFC-2119.

F1 Knowledge Base Management
F1.1User MUST be able to select/create the on-disk data folder.
F1.2User MUST be able to create, rename, duplicate, and delete profiles. Filenames are sanitized slugs; slug collisions de-duplicated (-2 suffix). Delete MUST confirm.
F1.3User MUST be able to paste free text → LLM structures it → diff/preview → save writes the .md.
F1.4User MUST be able to edit profile markdown directly in the side panel and externally (changes picked up on next read).
F1.5Tool MUST validate profile schema and warn on malformed sections without destroying content.
F1.6User MUST be able to select the active profile per application from the side panel.
F1.7User MUST be able to view/edit the Q&A bank in the side panel.
F2 Field Detection
F2.1On trigger, content script MUST scan the active tab for: <input>, <textarea>, <select>, radio/checkbox groups, contenteditable, and custom widgets (combobox, React-Select, date pickers).
F2.2For each field MUST capture: stable fieldId, visible label, name/id/autocomplete, type, options, required, and grouping.
F2.3MUST detect repeatable sections and the "add another" control (best-effort).
F2.4MUST detect file-upload fields and classify them (resume / cover letter / other).
F2.5MUST detect multi-step / wizard forms and the next/continue control (best-effort).
F3 Hybrid Field Mapping
F3.1Mapper MUST first apply deterministic rules (label/attribute/autocomplete heuristics).
F3.2Unmatched fields MUST be sent to the LLM with field metadata + relevant profile context.
F3.3Each result MUST carry source (rule|profile|qa|llm|blank) and confidence (high|medium|low).
F3.4Mapper MUST NEVER fabricate data not in the profile/Q&A bank. Unknown → blank + flag.
F3.5Values MUST be reformatted to match the target field's expected format.
F4 Custom Question Handling
F4.1For non-resume questions MUST first try to match a Q&A bank entry (rules then LLM semantic match).
F4.2On no match (if AI drafting on) MUST draft an answer grounded in profile + bank, flagged medium/low.
F4.3User MUST be able to edit any drafted answer and optionally save it back to the Q&A bank.
F4.4Tool MUST NOT answer with invented facts not derivable from the profile; flag for user instead.
F5 Fill & Review (Side Panel)
F5.1After mapping MUST show a review table: label, proposed value (editable), source, confidence, required flag, include/skip toggle.
F5.2Fields MUST be grouped to mirror the form (Personal, Experience #1, …).
F5.3User MUST be able to edit any value before applying.
F5.4"Apply to page" MUST write approved values using correct event dispatch (§11.7), reporting per-field success/failure.
F5.5Panel MUST clearly list flagged items (blanks, low-confidence, uploads, captcha).
F5.6Tool MUST NOT click the site's submit button. MAY offer "Mark as submitted" to update history.
F5.7User MUST be able to re-run mapping (e.g. after navigating to the next wizard step).
F6 File-Upload Flagging
F6.1Detected upload fields MUST be surfaced as flagged items with their detected purpose.
F6.2Tool MUST NOT set file inputs programmatically.
F7 Application History & Dedupe
F7.1On apply MUST record an entry in history.json.
F7.2Before/at fill time MUST check for duplicates (URL or company+role match) and warn.
F7.3User MUST be able to browse/search history and update status.
F8 LLM Provider Config & Fallback
F8.1User MUST be able to configure multiple providers with per-provider API key, model, and base URL.
F8.2User MUST be able to define an ordered fallback chain (default ≤3; supports N).
F8.3On provider failure MUST transparently try the next provider in order.
F8.4If all providers fail MUST show rules-only results with LLM-dependent fields flagged.
F8.5User SHOULD be able to test each provider's connectivity from settings.
F9 Settings & Onboarding
F9.1First-run onboarding MUST guide: pick data folder → create first profile → (optionally) add LLM provider. LLM step is optional — rules-only filling works with no provider.
F9.2Settings MUST cover providers/keys, fallback order, default profile, AI drafting on/off, confidence threshold, and data-folder reconnection.

Field Detection & Mapping Strategy

11.1 Detection

  • Walk DOM for fillable elements. Skip hidden/disabled/type=hidden and honeypots.
  • Field identity: each field gets a generated stable fieldId. Content script keeps an in-memory fieldId → live element map. Apply-to-page uses fieldId (not CSS selector) — re-querying at write time per EC-14.
  • Label priority: <label for> → wrapping <label> → aria-labelledby → aria-label → preceding text → placeholder → name/id (de-camelCased).
  • Record autocomplete tokens — high-signal for rules.

11.2 Rules Layer (Deterministic, Runs First)

Match on autocomplete tokens, then normalized label/name/id against a synonym dictionary. Output high-confidence mappings; everything else passes to LLM.

11.3 LLM Layer (Fallback)

  • Input: unresolved fields (label, type, options, group) + compact profile context + relevant Q&A entries.
  • Task: return per field a value drawn only from provided context + confidence; or blank if unknown.
  • Output: structured JSON (Appendix C) validated before use; invalid output → one repair retry → blank+flag.

11.4 Value Formatting

📅 Dates

Detect expected format from type=month/date, placeholder, or adjacent split selects. Reformat stored YYYY-MM accordingly. present/current → "currently work here" checkbox when available.

📋 Selects/Radios

Fuzzy-match stored value to closest available option (synonym table, e.g. "Bachelor's" ↔ "BSc" ↔ "Undergraduate"). No confident match → flag.

11.7 Writing Values into Live Forms (Critical)

Modern SPA forms ignore naïve element.value = x. The content script MUST use native setters and dispatch the right events.
  • Text inputs/textareas: set value via native setter (Object.getOwnPropertyDescriptor(HTMLInputElement.prototype,'value').set) then dispatch input and change events (bubbling) + focus/blur where needed.
  • <select>: set value/selectedIndex and dispatch change.
  • Checkboxes/radios: set checked and dispatch click/change.
  • Custom comboboxes: simulate focus → type → wait for option list → select via click/keyboard. Inherently fragile; failures reported per-field, not silently dropped.
  • Never write to disabled/readonly fields. Respect maxlength.

11.8 ATS Adapters

An adapter interface lets platform-specific modules override (not replace) detection, label-resolution, repeatable-section handling, and value-writing for Workday, Greenhouse, Lever, iCIMS, Taleo, and Bayt. Selected by URL/DOM fingerprint; absence falls back to the generic engine.

12

LLM Provider Layer

12.1 Abstraction

interface LLMProvider {
  id: string;   // "openai" | "gemini" | "anthropic" | "custom"
  name: string;
  complete(req: LLMRequest): Promise<LLMResponse>;
}
  • Thin adapters call provider REST endpoints via fetch — no heavy SDKs.
  • Normalize to common request (system + user prompt, JSON schema hint, low temperature) and response (text/JSON, token usage, error class).
  • A "custom" OpenAI-compatible provider covers self-hosted gateways and future local models — no new code required.
  • One model per provider (D22): single model used for both mapping and drafting. Default temperature: low. No separate model fields in v1.

12.2 Fallback Chain

  • Config: ordered list of providerIds, default length 3, supports N. Provider model/key/baseUrl come from per-provider config (§18) — the chain holds only the order.
  • Try provider[0]; on any failure advance to provider[1], etc.
  • Retryable: network error, 429, 5xx, timeout (30s), safety refusal, malformed output after one repair.
  • Config error (401/403): also advance to next provider and surface bad-key state in Settings (EH-3).
  • All fail → rules-only result + flagged fields + error toast (EH-1).

12.3 Cost & Token Control

Batch unresolved fields into as few calls as practical. Send compact profile context (only relevant sections). Expose token usage in debug log; respect a configurable max field batch size.

Security & Privacy

15.1
Local-first. All profile, Q&A, and history data stays on the user's disk. No cloud sync, no telemetry, no analytics, no external service other than the user-configured LLM providers.
15.2
Minimal LLM exposure. Only field metadata + relevant profile/Q&A snippets are sent over HTTPS. The job description is NOT sent (D15). PII transit to the chosen provider is disclosed in onboarding.
15.3
Secret storage. API keys live in chrome.storage.local (sandboxed), never in the data folder, never in the content script. Keys are only used in the background worker. Provide "clear keys" action. Document that chrome.storage.local is not encrypted at rest; offer an opt-in session-only key mode.
15.4
Content-script isolation. The content script holds no secrets and makes no network requests — only reads/writes the DOM on user action.
15.5
Least privilege. Use activeTab + on-demand scripting.executeScript for generic sites. Request host_permissions only for known-ATS adapter domains. No broad <all_urls> auto-injection.
15.6
No credential handling. The tool never reads, stores, or types login credentials — relies on the user's existing session.
15.7
Data portability/erasure. Data is plain files — the user can back up, move, or delete freely. Provide an in-app "delete all local data" that also clears keys and history.
16

Legal / Terms-of-Service

Assistive, not autonomous. The tool fills the form the user is already viewing and stops. It does not crawl, scrape at scale, mass-apply, navigate autonomously, or evade anti-automation measures.

LinkedIn excluded (N1). LinkedIn's ToS restrict automated interaction; Easy Apply automation is intentionally out of scope to protect the user's account.

Respect site terms. Known-ATS adapters operate within normal user interaction patterns. The tool must never bypass captchas/anti-bot systems.

Accuracy is the user's responsibility. The user reviews every value before submitting; the tool never fabricates facts. A disclaimer in onboarding states the user is responsible for the truthfulness of submitted data.

Appendix B — Internal Types (Illustrative)

type FieldType =
  | 'text' | 'textarea' | 'email' | 'tel' | 'url' | 'number'
  | 'date' | 'month' | 'select' | 'radio' | 'checkbox'
  | 'combobox' | 'file' | 'contenteditable' | 'unknown';

interface DetectedField {
  fieldId: string;      // generated stable id; key across detect→apply
  selector?: string;    // diagnostic only; may be ambiguous on dynamic pages
  label: string;
  name?: string; id?: string; autocomplete?: string;
  type: FieldType;
  options?: string[];   // for select/radio/combobox
  required: boolean;
  group?: string;       // e.g. "experience#2"
  maxLength?: number;
  isUpload?: boolean;
  uploadKind?: 'resume' | 'cover-letter' | 'other';
}

type MappingSource = 'rule' | 'profile' | 'qa' | 'llm' | 'blank';
type Confidence    = 'high' | 'medium' | 'low';

interface MappingResult {
  field: DetectedField;
  value: string | boolean | number | null;
  source: MappingSource;
  confidence: Confidence;
  needsReview: boolean;
  note?: string;    // e.g. "truncated", "no matching option"
  include: boolean; // user toggle (default = !needsReview)
}
